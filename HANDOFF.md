## What Is This? (Plain English)

ParBench is a research project that tests how well AI models (LLMs) can translate code between different parallel programming APIs — specifically CUDA, OpenMP, and OpenCL. Think of it like testing whether ChatGPT can convert a CUDA GPU program into an equivalent OpenMP CPU program that still produces the correct output.

We ran one AI model (Qwen 3.5 397B) through all our benchmarks. The data is on disk. Now we need to turn that raw data into **paper-ready artifacts**: a claims file that maps every number in the paper to its source data, publication-quality figures, and a model comparison table. We also need to update project documentation that became stale while the evaluation was running.

A second model (GPT-5.4) will be added later — this task is **Qwen-only**.

### What You're Building

1. **`paper_claims.json`** — A standalone file where every quantitative claim in the paper (e.g., "35.2% pass rate") maps to the exact data file, JSON path, and verification command that proves it. Think of it as a receipt for every number.

2. **Publication figures** (F2-F7, C.3-C.4) — Heatmaps, bar charts, degradation curves generated by matplotlib. These go into the LaTeX paper.

3. **Model comparison table** (T2) — A LaTeX snippet showing Qwen results per translation direction (GPT row will show "---" since we don't have that data yet).

4. **Documentation updates** — Several planning docs say "Phase 3 NOT STARTED" when the Qwen evaluation is actually complete. These need to reflect reality.

### What You're NOT Doing

- **Overleaf merge** — The user has a newer .tex version from teammates on Overleaf. They haven't downloaded it yet. Don't touch any .tex files.
- **GPT-5.4 evaluation** — Azure provisioning is still pending. Don't create or expect GPT result files.
- **Modifying result JSONs** — The 708 evaluation result files are immutable. Never edit or delete them.

## Glossary

| Term | Meaning |
|------|---------|
| **Canonical evaluation** | The main evaluation run: pass@3 (3 samples per task), L0 augmentation (no code perturbation), temp=0.7 |
| **Ablation** | Follow-up runs at augmentation levels L1-L4 on tasks that passed L0, testing robustness |
| **pass@1** | "At least 1 of 3 samples passes" — the primary success metric |
| **pass@3** | "All 3 of 3 samples pass" — stricter success metric |
| **KNOWN_FAIL** | 9 benchmark specs that are broken on our hardware (e.g., CUDA 12 removed `texture<>`) |
| **Campaign 1/2** | Legacy naming in `quantitative_findings.py`. Campaign 1 = temp=0.0 (empty). Campaign 2 = temp=0.7 (canonical). |
| **EXCLUDED_SPECS** | Python set in `harness/constants.py` with 9 KNOWN_FAIL spec IDs |
| **curated specs** | The ~87 specs included in eval batches (total 206 minus KNOWN_FAIL minus HeCBench bulk) |
| **omp_target** | OpenMP offload target — a separate API direction not included in the standard 6 translations |
| **Wilson CI** | Wilson score confidence interval — used for all pass-rate confidence bounds |
