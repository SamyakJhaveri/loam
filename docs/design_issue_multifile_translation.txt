  The issue I want to flag:

  When I dug into the 4 failures, 3 of 4 BUILD_FAILs are structural failures, not translation quality failures.

  The CUDA and OMP implementations in Rodinia were written independently. Their file structures reflect each developer's design choices — not a translation of each other. By asking the LLM to produce the exact Rodinia OMP file layout, we're simultaneously testing two orthogonal skills:

  1. Translation quality — does the LLM correctly map CUDA parallel patterns to OMP? ← what we intend to measure
  2. Code restructuring ability — can the LLM replicate a specific file organization that the OMP developer chose? ← a completely different skill

  Concrete examples:
  - kmeans (2 CUDA files → 8 OMP files across 2 subdirectories): The OMP version has both a kmeans_openmp/ variant and a kmeans_serial/ baseline in separate subdirs. The serial baseline has nothing to do with translating CUDA — it's a pre-existing reference. The LLM produced 1/8 files.
  - backprop (2 CUDA files → 4 OMP files): Different file factoring. The LLM translated the kernel but left 3 glue files empty, triggering a linker error.
  - streamcluster (2 CUDA files → 2 OMP files, correct count): Both files were produced, but the LLM defined a shared dist() helper in both, causing a redefinition error. File coordination failure.
  - srad (2 CUDA files → 1 OMP file — file structure correct): True translation error — the LLM passed argv[0] (a string) where argc (an int) was expected. Semantic mistake, not structural.

  Interestingly, lud PASSES despite having 6 files across 4 subdirectories — because all basenames are unique across dirs and the LLM correctly outputs full paths like filename=base/lud.c.

  The core question: is replicating the OMP reference file structure in scope for ParBench?

  Three options:

  My current lean: Option B in the paper (classify and report by complexity type) + Option C as a short-term technical fix (the extraction disambiguation bug is real and independent of the design question). The classification itself is a scientific contribution — it answers "where exactly
   do LLMs fail?" with a structural explanation.
                                                                                                                                                                                                                                                                                                 
  Expected PASS rates by class based on current data:                                                                                                                                                                                                                                            
  - 1:1_simple (3 kernels): ~100% — pure translation, no restructuring                                                                                                                                                                                                                         
  - N:1_reduction (5 kernels, bfs/hotspot/nw etc.): ~80% — LLMs handle file consolidation well                                                                                                                                                                                                   
  - N:M_reorganized (backprop, nn): ~25% — different factoring is hard                                                                                                                                                                                                                         
  - structural_complex (kmeans, lud, mummergpu...): ~20% — subdirs + duplicate basenames near-impossible zero-shot
