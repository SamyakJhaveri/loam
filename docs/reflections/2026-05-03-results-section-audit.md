## What Surprised Me

- **The figure tikz data includes KNOWN_FAIL tasks but the caption implies they're excluded.** The tikz file `f4_failure_taxonomy_qwen.tex` sums to 137 tasks, but after KNOWN_FAIL exclusion the 6 standard directions contain only 120 tasks. The 17-task gap is exactly the KNOWN_FAIL-involved tasks. This means the figure was generated from unfiltered data while the caption says "drawn from 626 valid records." This inconsistency between the figure generation pipeline and the exclusion logic used for statistics could have slipped into the final submission.

- **The stencil1d L4 translated code does NOT match the paper's claim.** The paper says L4 "reverts to CUDA-style tiling" but the actual code is a simple `#pragma omp parallel for` loop with a boundary condition bug (`idx < RADIUS` instead of `idx < 0`). Only L2 has CUDA-style tiling. This was a case where a plausible narrative ("augmentation causes regression to memorized CUDA patterns") was written without re-reading the actual translated code for each level.

- **Prose errors cluster around cross-model comparisons, not single-model statistics.** Every single-model number (pass rates, CIs, counts) was perfect. Errors appeared when comparing or summarizing across models: the augmentation range (picking L1 as lower bound instead of L4), the direction gap range ("2--10 pp" when two directions match exactly), and the pass@k paragraph attributing Qwen-specific numbers without model labels.

## Pattern Proposal

**Target:** `.claude/rules/known-issues.md` (append to existing section) or a new paper-audit checklist

```
## Paper Prose Verification Protocol

When auditing paper prose against data:
1. Single-model statistics: spot-check 2-3 against JSON source files (these are usually correct)
2. Cross-model comparisons: verify EVERY claim (ranges, gaps, rankings) -- errors cluster here
3. Per-kernel anecdotes: read the ACTUAL translated code, not just the overall_status field
4. Figure captions: verify task/record counts match the exclusion policy (KNOWN_FAIL, direction filter)
5. Statistical test labels: confirm the test name matches what the script actually runs (grep the source)
```

**Why:** This session found 5 factual errors, all in cross-model or narrative prose, zero in raw statistics. Future audits can be faster by focusing verification effort on the error-prone categories.

## Prompt Improvement

**Original approach:** "Read results.tex. Find out if everything is accurate."

**Better approach:** Separate the audit into two explicit passes with different verification strategies.

```
Pass 1 (table data): Spot-check 3 random cells per table against the source JSON. Tables are generated by scripts and are almost always correct.

Pass 2 (prose claims): For every sentence that compares models, states a range, or describes a specific kernel's behavior, verify independently. These are hand-written and error-prone. For kernel anecdotes, read the actual translated_files content, not just the status.
```

## Gotcha Discovered

**Symptom:** Paper claims Codex augmentation range is "86.6%--88.7%" but actual range is 85.6%--88.7%.
**Root cause:** The author likely looked at L1 (86.6%) and L2 (88.7%) as the range endpoints, but L4 (85.6%) is actually the minimum. When augmentation levels don't monotonically decrease, the min/max aren't at the extremes of the level sequence.
**Fix:** Always compute min/max across ALL levels when stating a range, not just eyeballing the first and last.
**Status:** NEW GOTCHA -- not previously documented. Specific to paper writing, not codebase.
