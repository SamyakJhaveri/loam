\begin{abstract}
Large language models are increasingly used to port code, but parallel code translation remains difficult to evaluate: repository-level studies conflate translation with build-system generation, while standard code benchmarks do not test GPU programming semantics or benchmark memorization. We present \parbench{}, a kernel-centric benchmark framework for LLM-based translation across CUDA, OpenMP, and OpenCL. ParBench combines declarative benchmark specifications, an end-to-end build-run-verify harness, survey-grounded corpus construction, and AST-driven semantics-preserving source augmentation. The corpus contains 96 specifications over 35 kernels from five HPC benchmark suites, yielding six standard translation directions plus two OpenMP-target case-study directions. Across 710 primary Qwen~3.5 397B-A17B tasks, ParBench observes a 38.3\% pass rate [34.8\%, 41.9\%], including 64.2\% [55.3\%, 72.2\%] on CUDA-to-OpenMP; GPT-4.1~mini reaches 30.7\% [27.1\%, 34.6\%] across 577 tasks. The dominant bottleneck is build-stage adaptation (33.9\% of all Qwen tasks), while verified semantic failures are smaller but non-negligible (7.2\%). Augmentation does not produce a monotonic degradation trend on the balanced CUDA-to-OpenMP subset ($z=0.0$, $p=1.0$), and pass@$k$ sampling rarely rescues hard failures. ParBench therefore exposes both non-trivial current capability and the specific structural failures that remain for reliable parallel code translation.
\end{abstract}

\section{Introduction}
\label{sec:introduction}

Scientific code increasingly needs to move across parallel programming APIs. CUDA implementations written for NVIDIA GPUs must be ported to CPU OpenMP, OpenCL, HIP, SYCL, OpenMP target offload, or other portability layers as hardware and procurement constraints change. Large language models (LLMs) are a plausible aid for this work, but parallel translation is not ordinary code translation. Correctness depends on preserving thread-index arithmetic, synchronization, memory-management behavior, reductions, address spaces, launch structure, and host-kernel interfaces.

Existing evaluation does not isolate this capability cleanly. Sequential code benchmarks such as HumanEval~\cite{HumanEval2021}, SWE-bench~\cite{SWEbench2024}, and TransCoder~\cite{TransCoder2020} do not evaluate parallel API semantics. Parallel generation benchmarks such as ParEval~\cite{ParEval2024} ask models to synthesize code from descriptions, not translate existing parallel structure. Repository-level translation, as in ParEval-Repo~\cite{ParEvalRepo2025}, evaluates a realistic workflow but makes build-system and integration generation the binding constraint: that study reports 0\% pass@1 on applications larger than 133~SLoC, including XSBench. This leaves a narrower and important question unanswered: can models translate the computational parallel kernel itself when the surrounding executable context is fixed? A fuller related-work comparison is in Appendix~\ref{sec:appendix-related}.

Benchmark contamination is a second challenge. HPC suites such as Rodinia~\cite{Rodinia2009}, XSBench~\cite{XSBench2014}, and HeCBench~\cite{HeCBench2023} are widely replicated, so success on their original source may reflect familiarity with public code rather than robust parallel reasoning. A benchmark for LLM parallel translation should therefore test executable correctness while also probing whether results survive semantics-preserving source variation.

We present \parbench{}, a kernel-centric framework for evaluating LLM-based parallel code translation. ParBench provides fixed build and verification infrastructure, asks the model to translate only the selected kernel files, and judges the output with a build-run-verify harness. A declarative specification describes each kernel/API variant; an AST-based augmentation engine creates transformed but equivalent source variants at five levels; and a unified evaluation pipeline records pass/fail outcomes and diagnostic failure categories. The benchmark corpus is not ad hoc: it is derived from a survey of 35 open-source HPC benchmark repositories and narrowed to suites with equivalent multi-API kernels and automated correctness checks.

This paper makes three contributions.

\begin{enumerate}
\item \textbf{A kernel-centric parallel translation benchmark.} ParBench provides 96 executable specifications spanning 35 kernels from Rodinia, HeCBench, XSBench, RSBench, and mixbench, with six standard directions across CUDA, OpenMP, and OpenCL plus two OpenMP-target case-study directions. Kernel isolation preserves executable evaluation while avoiding the build-system-generation confound identified by repository-level work.
\item \textbf{A spec-driven build-run-verify and augmentation framework.} ParBench uses declarative correctness contracts, a common harness, a failure taxonomy, and six AST-driven semantics-preserving transformations across levels L0--L4 to test robustness beyond surface-form familiarity. Full schema, prompt, and augmentation details appear in Appendix~\ref{sec:appendix-framework}.
\item \textbf{An empirical characterization of current LLM behavior.} On 1{,}287 primary tasks across Qwen~3.5 397B-A17B and GPT-4.1~mini, ParBench shows non-trivial translation capability under kernel isolation, build-stage adaptation as the dominant bottleneck, strong direction dependence, no monotonic augmentation degradation in the main trend test, and limited rescue from sampling alone.
\end{enumerate}

\section{ParBench Overview}
\label{sec:framework}

ParBench evaluates a translation task of the form: given source files implementing a kernel in one parallel API, produce target files implementing the same computation in another API so that the resulting program builds, runs, and passes the benchmark's correctness check. Figure~\ref{fig:architecture} summarizes the pipeline.

\subsection{Declarative Specifications}
\label{sec:spec-schema}

Each kernel/API variant is represented by a JSON specification that records identity, provenance, files, build commands, run arguments, and verification strategies. The key design choice is to separate \emph{translation targets} from support files: prompt payload files are shown to the model, target files identify what it must emit, support files and build infrastructure are fixed context, and verification-only files are never shown. This makes correctness definitions explicit and allows the same harness to evaluate every suite. The main verification strategies are exit-code checks and stdout-pattern matching, applied conjunctively; reserved schema fields support numeric comparison, file diffing, and custom scripts. A condensed spec example and schema details are in Appendix~\ref{sec:appendix-framework}.

\subsection{Kernel-Centric Evaluation}
\label{sec:eval-pipeline}

ParBench isolates kernel translation from repository reconstruction. The model receives the source kernel files, the target API, target filenames, the target build command, and read-only target infrastructure context needed to match signatures and data structures. Makefiles, shared headers, datasets, and reference outputs are not regenerated by the model. When source and target implementations use different command-line or stdout conventions, the evaluation uses source-side run arguments and accepts either source- or target-style verification patterns; this avoids penalizing correct translations for superficial interface formatting. The full prompt template and anonymization procedure are given in Appendix~\ref{sec:appendix-prompt}.

\subsection{Build, Run, Verify}
\label{sec:harness}

The harness applies three sequential stages. The build stage invokes the spec-defined build command and checks that the expected executable exists. The run stage executes the binary with the selected input configuration and records stdout, stderr, exit code, and timeout status. The verify stage applies all declared correctness checks. A task passes only if all stages succeed; otherwise it is labeled \buildfail{}, \runfail{}, \verifyfail{}, timeout, or \extractionfail{} if the LLM response cannot be parsed into target files. This taxonomy distinguishes compilation and API adaptation errors from translations that compile and run but produce incorrect output.

\subsection{Augmentation}
\label{sec:augmentation-engine}

To probe robustness to surface variation, ParBench augments source code through six libclang-backed transforms: condition swapping, compound-assignment expansion/contraction, local variable and parameter renaming, typedef expansion, pointer-arithmetic/indexing conversion, and static helper-function renaming. L0 is the original source; L1--L4 apply increasing transform and candidate-site density. Re-parsing guards against syntactically invalid rewrites. The augmentation baseline validates semantics preservation: 68 of 88 non-\knownfail{} specs across all five suites pass at every augmented level with zero correctness regressions. Transform definitions, level settings, and transform-frequency details appear in Appendix~\ref{sec:appendix-framework} and Appendix~\ref{sec:appendix-c3}.

\section{Benchmark Construction}
\label{sec:benchmark-curation}
\label{sec:suite-selection}

ParBench's corpus was assembled through a survey and filtering process rather than by convenience selection. We began with 40 candidate open-source HPC benchmark repositories from proxy-application catalogs, benchmark suites, and conference literature; 35 were analyzed after excluding download failures, duplicates, and insufficiently documented repositories. The survey covered suites, mini-applications, proxy applications, full applications, libraries, and microbenchmarks. Selection required open-source availability, equivalent implementations of the same kernel in multiple APIs, automated build/run/verify support, self-checking output, and domain diversity. Full survey figures, API co-occurrence data, and exclusion rationale are in Appendices~\ref{sec:appendix-a}--\ref{sec:appendix-b}.

The survey also motivates the API focus. CUDA-to-OpenMP provides the largest paradigm-crossing pool in the surveyed material, with 472 kernel-level CUDA--OpenMP pairs across six repositories. CUDA--HIP (633 pairs) and CUDA--SYCL (616 pairs) are larger by raw count, but HIP is close to syntactic renaming and SYCL coverage is concentrated in fewer suites; both are useful portability targets but less diagnostic for the structural question ParBench asks. OpenCL has much lower kernel-level coverage, yet it is retained because its separate host/kernel source model and address-space qualifiers expose a qualitatively different translation challenge. Appendix~\ref{sec:appendix-a} gives the full API co-occurrence analysis and exclusion rationale for HIP, SYCL, OpenACC, and OpenMP target offload.

The resulting corpus contains 96 specifications over 35 kernels (Table~\ref{tab:suite-summary}). Rodinia provides the main CUDA/OpenMP/OpenCL substrate with 22 kernels and 60 specs; 54 pass baseline verification and six are \knownfail{} due to toolchain or platform issues such as deprecated CUDA texture APIs, OpenGL dependencies, or pre-existing OpenCL runtime failures. HeCBench contributes 10 curated kernels selected from a much larger multi-API pool for build reliability and domain diversity. XSBench, RSBench, and mixbench add proxy-application and microbenchmark coverage beyond Rodinia.

The corpus spans 80--3{,}304 SLoC with a median of 271 SLoC, and 24 of 96 specs (25\%) require translating more than one source file. Thus kernel-centric evaluation does not reduce the task to toy functions: 31 of 35 kernels exceed the 133-SLoC scale at which ParEval-Repo reports repository-level failure~\cite{ParEvalRepo2025}. \section{Experimental Setup}
\label{sec:experimental-setup}

We evaluate two provider-diverse models: Qwen~3.5 397B-A17B through Together AI and GPT-4.1~mini through Azure OpenAI. The primary campaigns use temperature~0, no chain-of-thought mode, and up to three self-repair attempts after the initial translation. A separate pass@$k$ sweep for Qwen~3.5 uses temperature~0.7, L0 only, three independent samples, and no self-repair. Model and hardware details are in Appendix Tables~\ref{tab:model-config} and~\ref{tab:hardware}; token and cost accounting is in Appendix~\ref{sec:appendix-cost}.

The standard evaluation matrix covers six directions: CUDA$\leftrightarrow$OpenMP, CUDA$\leftrightarrow$OpenCL, and OpenMP$\leftrightarrow$OpenCL. Two OpenMP-target directions are included as case studies because they are available only in a smaller subset and require a different compiler stack. For Qwen~3.5, the primary campaign contains 142 L0 source-target pairs across all directions and five augmentation levels, for 710 tasks. GPT-4.1~mini covers 577 tasks across seven directions and five augmentation levels.

The primary metric is greedy pass rate: the fraction of tasks that pass build, run, and verify after deterministic decoding and bounded self-repair. We report Wilson 95\% confidence intervals for proportions. Direction asymmetry is tested with McNemar's exact test on paired kernels; augmentation trend is tested with Cochran-Armitage on the balanced CUDA-to-OpenMP subset; and cross-model aggregate differences are tested with chi-squared tests. Performance speedup is intentionally out of scope: wall-clock measurements in the current harness conflate kernel runtime with I/O, allocation, launch overhead, and scheduling noise.

Kernel-centric evaluation reveals capability that repository-level evaluation hides. \subsection{File Coordination Effects}
\label{sec:file-coordination}

Kernel isolation removes repository build generation from the task, but it does not remove cross-file reasoning. Some ParBench translations still require coordinated edits across host wrappers, kernel files, and helper interfaces. Table~\ref{tab:file-complexity} shows that this dimension strongly affects Qwen~3.5 outcomes: single-file translations pass at 51.3\%, while multi-file classes range from 36.3\% down to 8.0\%. A chi-squared test over the four file-cardinality classes yields $\chi^2=82.73$ ($df=3$, $p=0.005$). Thus, multi-file coordination is a separate bottleneck from build-system generation: even when the build scaffold is fixed, the model must keep declarations, launch sites, and function signatures consistent across emitted files.

\subsection{Failure Modes and Self-Repair}
\label{sec:failure-taxonomy}

\begin{figure}[t]
\centering
\includegraphics[width=\columnwidth]{f4_failure_taxonomy_qwen}
\caption{Qwen~3.5 outcome taxonomy across 710 primary tasks. Build-stage failures are the largest non-pass category. GPT-4.1~mini failure visualizations are in Appendix~\ref{sec:appendix-gpt-figures}.}
\label{fig:failure-taxonomy}

Build-stage adaptation is the dominant bottleneck. In the Qwen campaign, \buildfail{} accounts for 241/710 tasks (33.9\%) and 55.0\% of all failures. The largest build-failure subcategories are undeclared identifiers (56 tasks), missing headers (55), and linker errors (49). These often indicate incomplete API-surface adaptation: retained CUDA identifiers in OpenMP translations, missing OpenCL address-space qualifiers, or mismatched target-side dependencies.

\verifyfail{} is smaller but important: 51/710 tasks (7.2\%) compile and run but fail correctness. Of these, 46 produce wrong numerical output and 5 produce missing output. These cases are the strongest evidence that compilation alone is insufficient: exit-code-only or compile-only checks would misclassify them as successful translations. Detailed examples include OpenCL vector type conflicts in \texttt{cfd}, thread mapping errors in \texttt{hotspot}, and output-structure mistakes in \texttt{gaussian}; these case studies appear in Appendix~\ref{sec:appendix-c}.

Self-repair improves results but does not solve the task. For Qwen~3.5, the first-attempt pass rate is 22.5\% [19.6\%, 25.8\%]; after up to three diagnostic retries, it rises to 38.3\% [34.8\%, 41.9\%], a 15.8 percentage-point gain and 70.0\% relative increase. Of 550 initially failing tasks, 112 are repaired, while seven regressions occur. Compiler feedback is most useful for build errors, whereas runtime and verification failures offer weaker diagnostic signals. Transition matrices and direction-level repair rates are in Appendix Tables~\ref{tab:repair-transitions}--\ref{tab:self-repair}.

\subsection{Augmentation Robustness and Sampling}
\label{sec:augmentation-robustness}

Augmentation does not show the monotonic degradation expected if baseline success were mainly driven by exact surface familiarity. On the balanced CUDA-to-OpenMP subset ($n=24$ kernels per level), Qwen pass counts are 16, 14, 17, 14, and 16 at L0--L4. The Cochran-Armitage trend test yields $z=0.0$, $p=1.0$. This is a null result, not proof of invariance: with $n=24$ per level, the test has 80\% power only for monotonic trends above roughly 34 percentage points. Still, the observed stability argues against a simple memorized-source explanation. Full augmentation rates and heatmaps appear in Appendix Table~\ref{tab:augmentation-rates} and Figure~\ref{fig:aug-heatmap}.

Sampling also has limited effect. In the Qwen pass@$k$ sweep (142 L0 direction-kernel pairs, three samples each at temperature~0.7, no self-repair), 103 pairs (72.5\%) are hard failures with no passing sample, 22 (15.5\%) are partial successes, and 17 (12.0\%) pass in all samples. No pair has pass@1 equal to zero but pass@3 above zero. Additional samples therefore do not dramatically rescue the task; many failures appear structural at the kernel-direction level. Full pass@$k$ results are in Appendix Table~\ref{tab:pass-at-k}.

\section{Related Work}
\label{sec:related-work}

ParBench sits at the intersection of code translation benchmarks, LLM-for-HPC evaluation, and robustness testing. General-purpose code benchmarks have established execution-based evaluation as the standard for measuring functional correctness, but their task distributions are far from parallel HPC translation. HumanEval~\cite{HumanEval2021} and TransCoder~\cite{TransCoder2020} operate at function granularity, while SWE-bench~\cite{SWEbench2024} evaluates repository-level issue resolution in general software projects. These benchmarks are valuable, but they do not exercise GPU memory models, thread synchronization, API-specific launch structure, or host-device coordination.

Parallel-code benchmarks are closer but still answer different questions. ParEval~\cite{ParEval2024} studies synthesis of parallel code from natural language tasks, which tests whether a model can produce a solution but not whether it can preserve an existing implementation's decomposition across APIs. ParEval-Repo~\cite{ParEvalRepo2025} studies full-repository translation and demonstrates a severe integration bottleneck: models fail once they must recreate build systems and repository scaffolding. ParBench is complementary to both. It preserves executable evaluation but fixes build infrastructure so the measured object is kernel translation.

LASSI~\cite{LASSI2024} is the nearest prior work in granularity and verification. It evaluates an agentic self-correction pipeline for CUDA/OpenMP-target translation on 10 HeCBench kernels, whereas ParBench evaluates a broader survey-grounded corpus, six standard directions, bounded self-repair, and augmentation robustness. CodeRosetta~\cite{CodeRosetta2024} and HPC-Coder-V2~\cite{HPCCoderV2} demonstrate specialized modeling and fine-tuning for parallel code, but ParBench provides an execution-based substrate for comparing such models against general-purpose LLMs. Extended comparisons are in Appendix~\ref{sec:appendix-related}.

Finally, ParBench follows recent concerns about benchmark validity. Public code benchmarks can be memorized, and surface-form perturbations can reveal when evaluation depends on brittle pattern matching. ParBench's augmentation engine is not a complete solution to training-data contamination because it cannot rule out algorithm-level familiarity, but it provides a concrete robustness probe for widely circulated HPC kernels. Its correctness-only scope also differs from efficiency-oriented work such as TRACY~\cite{TRACY2025}, which shows that functional correctness and performance are separable.

\section{Discussion, Limitations, and Conclusion}
\label{sec:discussion}

ParBench establishes that LLMs can perform non-trivial parallel code translation when evaluation isolates the kernel and preserves executable correctness. It also shows where current systems fail. The main obstacle is not only semantic reasoning over parallel computation, nor only build syntax, but a spectrum of incomplete API adaptation, host-kernel coordination, and genuine correctness errors. Direction-level results matter: aggregate pass rates conceal that CUDA-to-OpenMP is much more tractable than OpenCL-to-CUDA, and that model comparisons differ by direction rather than uniformly across the benchmark.

The benchmark is correctness-focused. It does not measure speedups, occupancy, memory bandwidth, or efficiency; correct translated code may still be unusably slow, a concern also raised by TRACY~\cite{TRACY2025}. The corpus is also finite. Some directions have small paired samples, the augmentation trend test is underpowered for moderate effects, and several specs are marked \knownfail{} because of platform or toolchain constraints. Rodinia and related suites are likely present in training data, and augmentation tests surface-form robustness rather than deeper algorithmic memorization. Finally, the experiments cover two models and one bounded self-repair protocol; broader model families, agentic systems such as LASSI~\cite{LASSI2024}, fine-tuned HPC models~\cite{HPCCoderV2}, and efficiency-aware pipelines remain important follow-up evaluations.

ParBench is intended as evaluation infrastructure rather than a new translation method. Its contribution is to make the measurement problem sharper: kernel-centric isolation avoids repository-level build confounds, declarative specs make correctness reproducible, augmentation probes robustness, and the harness exposes where translations fail. The initial results show that current LLMs have real but uneven parallel translation capability, with build adaptation, direction asymmetry, and multi-file coordination as central barriers to reliable use.
