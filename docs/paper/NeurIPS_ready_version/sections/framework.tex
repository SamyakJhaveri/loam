\section{ParBench Framework}
\label{sec:framework}

ParBench is a benchmark framework designed to evaluate LLM-based parallel code translation through automated, end-to-end correctness assessment. The framework comprises four integrated components: (1)~a declarative spec schema that encodes what constitutes correct execution for each benchmark kernel, (2)~a three-stage build/run/verify pipeline that evaluates translated code, (3)~an AST-driven augmentation engine that generates semantically-equivalent code variants to assess translation robustness in the face of code perturbations, and (4)~an evaluation pipeline that orchestrates LLM-based translation and records diagnostic failure categories. This section describes each component in detail.

\subsection{Spec Schema: Declarative Correctness Contracts}
\label{sec:spec-schema}

Each benchmark in ParBench is described by a JSON specification (spec) that serves as a declarative contract defining what ``correct execution'' means for that kernel--API variant. As illustrated by the spec configuration flow in Figure~\ref{fig:architecture} (stage~1$\rightarrow$3 and stage~1$\rightarrow$4), the spec provides both prompt configuration to the Evaluation Pipeline and build/run/verify configuration to the Harness. The spec encodes all information required to build, run, and verify a benchmark without manual intervention, enabling fully automated evaluation. The schema (version 1.0.0) is formally defined in JSON Schema draft-07 and enforced by an offline validator.

A spec is organized into the following field groups:

\textbf{Identity and provenance.} The \texttt{identity} block assigns a globally unique identifier following the convention \texttt{\{source\_suite\}-\{kernel\_name\}-\{parallel\_api\}} (e.g., \texttt{rodinia-bfs-cuda}), and records the parallel API from a controlled vocabulary of 15 APIs including CUDA, OpenMP, OpenCL, HIP, SYCL, and serial. The \texttt{provenance} block pins the exact source repository and commit hash, ensuring reproducibility across environments.

\textbf{File partitioning.} The \texttt{files} block partitions benchmark source files into three disjoint sets: \texttt{prompt\_payload} (files provided to the LLM during translation), \texttt{support\_files} (build infrastructure such as Makefiles and shared headers --- contents of the header are provided as read-only reference in the translation prompt), and \texttt{verification\_only} (reference implementations that are never provided to the LLM). The fourth field, \texttt{translation\_targets}, identifies a subset of \texttt{prompt\_payload} that the LLM must translate. This field is central to the kernel-centric translation methodology described in Section~\ref{sec:eval-pipeline}.

\textbf{Build, run, and verification.} The \texttt{build} block specifies the build system, compiler commands, environment variables, and expected executable path. The \texttt{run} block defines named input configurations (e.g., \texttt{correctness}, \texttt{performance}) with arguments, timeout, and optional environment variables. The \texttt{verification} block declares an ordered list of verification strategies---evaluated in conjunction so that all must pass for the overall verdict to be PASS (see Section~\ref{sec:harness} for evaluation semantics). Five strategy types are currently implemented: \texttt{exit\_code} (check process return code), \texttt{stdout\_pattern} (regex match against captured standard output), \texttt{stdout\_exclude\_pattern} (reject output containing a failure signature), \texttt{numeric\_comparison} (extract a scalar from stdout and compare against a reference within a configurable tolerance), and \texttt{file\_hash} (SHA-256 of a designated output file must match a reference hash). Two additional types (\texttt{file\_diff}, \texttt{custom\_script}) are reserved in the schema for future work.

This design separates the definition of correctness from the mechanism of verification: the harness can evolve independently of what ``correct'' means for each kernel. For example, the \texttt{main} function that typically launches the kernel may contain an output verification check and generate return code depending on the pass or fail scenario. We have defined 96 such specs spanning five benchmark suites: 60 from Rodinia~\cite{Rodinia2009}, 4 from XSBench~\cite{XSBench2014}, 4 from RSBench, 3 from mixbench, and 25 from a curated subset of HeCBench~\cite{HeCBench2023}. Of these, 87 pass the baseline build/run/verify pipeline on the evaluation platform; the remaining 9 (\knownfail{} specs) fail due to platform-specific issues---CUDA~12 API deprecations, missing system libraries, or pre-existing runtime errors---and are excluded from evaluation (Section~\ref{sec:benchmark-curation}).

\textbf{Baseline results.} The \texttt{baseline\_results} block, populated by running the original implementation on a reference platform, records the expected output for each input configuration. This provides the ground truth to evaluate the functional correctness of the translated code.

\subsection{Evaluation Pipeline: Kernel-Centric Translation}
\label{sec:eval-pipeline}

The evaluation pipeline --- corresponding to the Evaluation Pipeline component in Figure~\ref{fig:architecture} (blue, center) --- orchestrates LLM-based parallel code translation through a structured prompt-response-verify cycle. Its design reflects a deliberate methodological choice: kernel-centric translation, in which only the parallel computation files are translated while the surrounding build infrastructure remains untouched.

\textbf{Kernel-centric methodology.} ParBench isolates the translation skill by feeding only the \texttt{translation\_targets} files (the parallel kernel code) to the LLM; Makefiles, shared headers, serial baselines, and utility code remain unchanged in the target directory. This design enables evaluation of whether an LLM can correctly map parallel programming patterns from one API to another, independent of its ability to replicate project file organization.

\textbf{Prompt structure.} Each translation prompt consists of a system message establishing the role of a parallel programming expert and a user message containing: (1)~the source kernel code with API-specific syntax highlighting, (2)~the target API and the filenames the LLM must produce, (3)~the target build command for compilation compatibility, and (4)~read-only target infrastructure context (non-kernel files from the target directory) so the LLM can match expected function signatures and data structures. Source support headers are included with instructions to inline definitions rather than emit unresolvable \texttt{\#include} directives. The full prompt template is provided in Appendix~\ref{sec:appendix-prompt}.

\textbf{Cross-API argument and verification handling.} When translating between APIs (e.g., CUDA to OpenMP), the source and target implementations may expect different command-line arguments and produce different \texttt{stdout} formats. ParBench handles this asymmetry with two branches. For \emph{full-program} translations (e.g., CUDA to OpenMP), the pipeline uses the \emph{source} spec's run arguments because LLMs preserve the source implementation's \texttt{argc}/\texttt{argv} parsing convention---a translated binary retains the argument-handling logic of the code it was translated from, not the reference target. For verification, the pipeline constructs a union of source and target \texttt{stdout\_pattern} regexes via alternation (\texttt{source\_pattern|target\_pattern}), accommodating translations that produce either output format. For \emph{kernel-only} translations (e.g., translations targeting OpenCL, where only \texttt{.cl} kernel files are rewritten and the host binary is unchanged), the pipeline uses the \emph{target} spec's run arguments and verification patterns directly, since the host program's argument parsing and output format are unmodified. This dual handling ensures that the evaluation measures translation quality rather than penalizing correct translations for superficial format differences.

ParBench distinguishes \emph{kernel-only} translations --- where only the device kernel files are translated while the host code remains untouched --- from \emph{full-program} translations, where the LLM rewrites all source files, including host-side API calls. The distinction arises naturally from API architecture: translations targeting OpenCL produce only \texttt{.cl} kernel files, because the OpenCL programming model separates kernel source (compiled at runtime via \texttt{clBuildProgram()}) from host orchestration code. In contrast, CUDA-to-OpenMP translations require rewriting the entire source file, replacing device kernels, memory management, and launch syntax with OpenMP pragmas and CPU-side equivalents. ParBench detects kernel-only translations automatically by checking whether all \texttt{translation\_targets} in the target spec end with \texttt{.cl}. For kernel-only translations, the target spec's native run arguments and verification patterns are used directly (since the host binary is unchanged), whereas full-program translations use the cross-API argument and verification handling described above.

To guard against such false positives, the verification pipeline includes a post-verification scan of \texttt{stdout} for OpenCL kernel build failure signatures. Translations that pass exit-code and stdout-pattern checks but contain evidence of kernel compilation failure are downgraded from PASS to \verifyfail{}. This specificity filter reduces false positives with zero additional false negatives, since legitimate kernel executions do not produce \texttt{clBuildProgram()} error messages.

\textbf{Optional self-repair.} The framework supports an iterative self-repair path (dashed arrow in Figure~\ref{fig:architecture}): on failure, the pipeline can feed failure-specific diagnostics back to the LLM for a corrected translation attempt. This capability is not exercised in the evaluation reported here; each sample is a single-attempt translation with no iterative repair (Section~\ref{sec:experimental-setup}).

\textbf{Response extraction.} The evaluation pipeline parses each LLM response to recover the translated source files using a four-tier fallback strategy. Tier~1 matches fenced code blocks annotated with an explicit \texttt{filename=} tag (the format requested by the system prompt). Tier~1.5 handles a space-separated variant where the filename follows the language identifier without a \texttt{filename=} prefix. Tier~2 applies fuzzy matching, associating a code block with a target filename mentioned within 500~characters preceding the fence. Tier~3 applies only when exactly one target file and one code block remain, assigning the block by elimination. If any expected target file cannot be recovered after all tiers, or if an extracted file is empty, the task is classified as \extractionfail{}. The genericized target filenames (\texttt{translated\_0.ext}, etc.) established during prompt anonymization (Appendix~\ref{sec:appendix-prompt}) are mapped back to the original filenames before the extracted files are written to disk for harness evaluation.
  
\textbf{Complexity taxonomy.} To enable stratified analysis of translation difficulty, each source--target pair is classified into one of four complexity classes based on the file cardinality of the translation: \texttt{single\_file} (1 source file to 1 target file), \texttt{multi\_to\_single} ($N$ source files to 1 target file), \texttt{single\_to\_multi} (1 source file to $N$ targets), and \texttt{multi\_to\_multi} ($N$ source files to $M$ target files).

\subsection{Augmentation Engine: Probing Robustness via Code Perturbations}
\label{sec:augmentation-engine}

Because widely used suites such as Rodinia~\cite{Rodinia2009} may already appear in LLM training data, the augmentation engine---the Augmentation component in Figure~\ref{fig:architecture} (gold, top-center)---generates semantically-equivalent code variants by applying AST-level transformations.

The engine implements six transforms, each backed by the libclang API for AST analysis to guarantee syntactic validity:
\begin{enumerate}
    \item \textbf{SwapCondition}---reverses operands of comparison expressions (e.g., \texttt{a < b} becomes \texttt{b > a}), with guards against side-effect-bearing subexpressions and assignments.
    
    \item \textbf{ArithmeticTransform}---converts between compound and expanded assignment forms (e.g., \texttt{x += 1} becomes \texttt{x = x + 1} and vice versa). 
    
    \item \textbf{ChangeNames}---consistently renames local variables and function parameters using symbol-aware reference collection, respecting scope boundaries and OpenMP pragma clauses.
    
    \item \textbf{TypedefExpansion}---replaces typedef aliases with their underlying primitive types at AST-identified type-reference sites.
    \item \textbf{PointerArithmeticToArrayIndex}---converts between pointer dereference and array indexing notation (e.g., \texttt{*(arr + i)} becomes \texttt{arr[i]}).
    
    \item \textbf{ChangeFunctionNames}---renames \texttt{static} (internal-linkage) helper functions across their definition and all call sites within the translation unit. \texttt{\_\_global\_\_} and \texttt{\_\_kernel\_\_} entry points are excluded because OpenCL resolves kernel names at runtime via string lookup in \texttt{clCreateKernel}; renaming the C function without updating the corresponding string literal would silently break execution.
\end{enumerate}

These six transforms target surface-level code features---naming conventions, syntax sugar, type aliases, and expression forms---that are most likely to be memorized verbatim by LLMs trained on open-source repositories, while preserving the parallel structure (thread indexing, synchronization, memory management) that is the subject of evaluation. Parallelism-aware transforms such as loop reordering or reduction-strategy substitution are out of scope because they risk altering the observable output and would require per-kernel correctness proofs.

All six transforms apply to every source file in the prompt payload---including both host code and GPU kernel function bodies. AST nodes originating from \texttt{\#include}d headers are excluded; only symbols defined in the main translation unit are rewritten. All transforms operate on the parsed AST rather than on raw text, which avoids the fragility of regex-based rewriting. Each candidate rewrite is validated by re-parsing the transformed code and confirming that no new diagnostic errors are introduced.

Five augmentation levels (L0--L4) control the density of applied transforms. At L0, the source code is unmodified. At L1, exactly one transform is selected and applied to a single candidate site. \subsection{Harness Pipeline: Build, Run, Verify}
\label{sec:harness}

The harness pipeline is a three-stage evaluation engine invoked via a command-line interface, corresponding to the Build, Run, and Verify sub-stages within the Harness Pipeline component of Figure~\ref{fig:architecture} (light blue, center-right). Each stage consumes the output of the previous stage and failure at any stage short-circuits subsequent stages. The Harness Pipeline receives translated files from the Evaluation Pipeline and build/run/verify configuration from the Spec JSON, and then emits a verdict to the Result node.

\textbf{Build stage.} The builder resolves the spec's working directory, executes an optional \texttt{clean} command, runs any \texttt{configure} step, substitutes template variables (e.g., \texttt{\$\{CUDA\_DIR\}}) from a platform-specific configuration file that maps tokens to local compiler and SDK paths, and invokes the build via a shell subprocess. After compilation, the builder verifies that the expected executable exists as a file on the disk. The build stage captures both \texttt{stdout} and \texttt{stderr} for diagnostic reporting and enforces a configurable timeout (default: 600 seconds).

\textbf{Run stage.} The runner executes the compiled binary with arguments drawn from the spec's named input configuration (typically \texttt{correctness}). Execution proceeds without a shell to avoid argument-parsing ambiguities. The runner captures \texttt{stdout}, \texttt{stderr}, exit code, and wall-clock duration, and enforces the spec-defined timeout (default: 300 seconds). An optional CPU-time measurement mode wraps execution with \texttt{/usr/bin/time -v} to capture user+system time independently of wall-clock duration.

\textbf{Verify stage.} The verifier applies the spec's verification strategies in declared order. For \texttt{exit\_code} verification, the actual exit code is compared against the expected value. For \texttt{stdout\_pattern} verification, a regular expression is applied to the captured stdout. A FAIL or ERROR result triggers immediate termination; a PASS is recorded and evaluation continues to the next strategy---all implemented strategies must pass for the overall verdict to be PASS. In cross-API translation evaluation, the verification stage uses a combined pattern that accepts output matching either the source or target spec's \texttt{stdout} format (Section~\ref{sec:eval-pipeline}), preventing false negatives from format differences between API implementations. 

The conjunction of exit code and \texttt{stdout}-pattern verification is essential for accurate measurement. For example, a CUDA translation of the Gaussian elimination kernel (from OpenCL) compiles without error, runs to completion with exit code zero, and produces partial \texttt{stdout} output --- matrix creation messages appear normally --- but fails to print the expected \texttt{Total:} timing summary that the reference implementation produces. This indicates a systematic translation defect rather than a transient failure. Compilation-only verification~\cite{CodeRosetta2024} would misclassify such a task as a successful translation; exit-code-only verification would similarly fail, since this translation terminated normally without signaling an error.

Each harness invocation produces one of four failure classifications---BUILD\_FAIL, RUN\_FAIL, VERIFY\_FAIL, or TIMEOUT---providing diagnostic granularity that enables systematic failure-mode analysis. The evaluation pipeline (Section~\ref{sec:eval-pipeline}) adds a fifth classification, EXTRACTION\_FAIL, when the LLM response does not contain parseable target files. % A \texttt{--resume} flag allows batch evaluations to skip previously completed tasks, supporting append-only result accumulation without re-executing passing benchmarks.
