Loam v3.0 — Architecture Approaches

Three ways to restructure the template for elegance and maintainability

A

Clean Subdirectory (Copier + Isolation)

Move all deliverables into seed/. Copier's _subdirectory renders only that tree — zero exclusion patterns needed. Template machinery lives outside and is invisible to Copier.

Repository Layout
seed/ ← Copier renders THIS
.claude/skills/ (curated ~15)
.claude/agents/ (11 agents)
.claude/rules/ (JVC routing)
.claude/hooks/ (pipeline gate)
CLAUDE.md.jinja
AGENTS.md.jinja
README.md.jinja

cultivation/ ← Skill dev & testing
experimental/
research-overlay/
marketplace/

soil/ ← Knowledge & learning
jvc-vault/
playbooks/

bin/ docs/ copier.yml

Pros

  • Zero exclusion patterns (vs current 36)
  • Keeps Copier's update mechanism
  • Clear visual separation
  • Symlink .claude/ → seed/.claude/ for dev

Cons

  • Still depends on Copier
  • Symlink adds dev setup step
  • Jinja rendering still required
B

Loam CLI (Custom Delivery)

Replace Copier with a thin loam CLI. Uses git mechanisms under the hood. loam init, loam sync, loam promote. Full control over the delivery pipeline.

Repository Layout
blueprint/ ← What gets delivered
.claude/ (full config)
CLAUDE.md.tmpl
AGENTS.md.tmpl
loam.yml (project config)

cli/ ← The loam tool
init.sh
sync.sh
promote.sh

lab/ ← Development area
skills-wip/
research/

knowledge/ ← JVC + reference
docs/

Pros

  • Full control over delivery pipeline
  • Git-native sync (subtree/cherry-pick)
  • No Copier dependency
  • Simpler template syntax (no jinja)

Cons

  • Must build + maintain a CLI
  • Loses Copier's 3-way merge on updates
  • More upfront work
C

Composable Layers (Pick Your Level)

Separate the deliverable into tiers: Foundation (always), Skills (grouped), Research (opt-in). Any delivery mechanism works. Most flexible.

Repository Layout
foundation/ ← Always installed
CLAUDE.md, AGENTS.md
.claude/rules/ (JVC routing)
.claude/hooks/ (pipeline gate)
.claude/settings.json

layers/ ← Opt-in groups
workflow/ (commit,pr,validate)
quality/ (review,critique)
planning/ (feature-dev,spec)
memory/ (dream,reflect)
research/ (paper,experiment)

workshop/ ← Development
knowledge/ ← JVC + reference
docs/

Pros

  • Users pick only what they need
  • Delivery-mechanism agnostic
  • Clear skill grouping
  • Easy to add new layers

Cons

  • Most complex to implement
  • Layer interdependencies need management
  • Settings.json merging across layers is tricky
