Failure Root Cause Analysis

Phase 1 (10-kernel batch, 2026-03-19): 4 BUILD_FAILs decompose into two fundamentally different categories.

3
STRUCTURAL FAILURE — File structure mismatch between source and target spec
Kernels: backprop (1→4 files), kmeans (2→8 files across 2 subdirs w/ duplicate basenames), streamcluster (2→2 files with shared helper redefinition)
Root cause: LLM is asked to replicate Rodinia OMP reference file organization, which was written independently from the CUDA source. Design decision needed — see Design Concern section.
1
SEMANTIC FAILURE — Correct file extracted, wrong code generated
Kernels: srad (1→1 file, but LLM passes argv[0] where argc expected — type error; Attempt 1 compiled, Attempt 2 re-broke it)
Root cause: LLM introduced a parameter type mismatch between function definition and call site. Error feedback loop failed — Attempt 2 "fixed" differently and broke the build.

⚠ Design Concern: Source vs. Target File Structure Mismatch

The fundamental issue: For multi-file benchmarks, the CUDA and OMP implementations in Rodinia were written independently by different people. Their file structures reflect API-specific design choices, not translations of each other. When we ask an LLM to translate CUDA→OMP and produce the OMP reference's exact file layout, we are testing two orthogonal skills simultaneously:
  1. Translation quality — does the LLM correctly map CUDA parallel patterns to OMP?
  2. Code restructuring ability — can the LLM replicate a specific file organization that the original developer chose?
These are very different things. A failure at skill 2 does not mean the LLM can't translate the algorithm.

Note on Speedup Ratio

The speedup ratio for bfs CUDA→OMP is ~0.001–0.002×, meaning OpenMP is ~500–1000× slower than CUDA for BFS. This is expected — BFS on GPU has massive parallelism advantages.

The speedup metric here measures translation correctness (does the LLM produce code that builds, runs, and passes verification?), not API performance. A ratio < 1.0 is a valid result: the LLM preserved correctness, but the target API is inherently slower for this workload on this hardware.

Timing method note: Claude bfs used cpu_time (via GNU time, user+sys=0.83s); Azure bfs used wall_time (0.482s). The baseline is always wall_time from the spec, creating a slight apples-to-oranges comparison. Re-running all baselines with CPU timing is a future task.
