Design Concern: Multi-File Structural Mismatch

3 of 4 BUILD_FAILs arise not from incorrect translation logic but from a mismatch between how CUDA and OpenMP variants of Rodinia are structured. Some benchmarks have 2 CUDA files but 4 OMP files (different authors wrote them independently). The LLM is being asked to replicate a target file structure it was never shown. This is an active design discussion — see docs/design_concern_multifile_translation.md for the full analysis and proposed solutions.

Self-Repair (Retry) Analysis

Failed build or run output is fed back to the model for a second attempt.

— Tasks where retry improved result (azure-gpt-4.1) Pending per-model retry analysis for azure-gpt-4.1 (10 tasks)
— Tasks passing on first attempt only Retry analysis will be updated once per-attempt data is extracted

Methodology: max_retries=2 — failed builds and runs are retried once. On retry, the error output (compiler stderr, run stderr, or diff) is included in the next prompt alongside the original code, giving the model diagnostic context for self-correction. A task is counted as “repaired” if the final attempt status is better than the first (e.g., BUILD_FAIL → PASS).
