ParBench System Architecture

AST-driven augmentation + LLM evaluation pipeline for parallel benchmark translation

1. Project Directory Structure

specs/

184 specs: 60 Rodinia + 120 HeCBench + 4 XSBench

  • 23 CUDA 21 OMP 21 OpenCL + HeCBench
  • Format: {suite}-{slug}-{api}.json
  • Each defines build, run, verify, and file info

c_augmentation/

Core transform engine (libclang-backed)

  • augment_dataset.py — 6 transforms + validation
  • test_transforms.py — unit tests (incl. Bug A/B/C regression)
  • Uses clang.cindex for AST analysis
  • LEVEL_FRACTIONS + _greedy_valid_subset() for robust composition
  • Supports .cu / .cpp / .c / .cl files

harness/

Build → Run → Verify pipeline

  • builder.py — clean/configure/build/verify-exe
  • runner.py — execute with timeout
  • verifier.py — exit_code / stdout_pattern
  • spec_loader.py — path resolution + prompt payload

scripts/augmentation/

Orchestration layer

  • augment_verify.py — single-spec pipeline
  • run_augment_batch.py — batch runner (JSON+MD)
  • combine_aug_results.py — merge reports

results/augmentation/

Evaluation output

  • JSON files with per-run details
  • Markdown reports with result matrices
  • Organized by eval run (eval_cuda, eval_omp, eval_opencl)

config/ & schema/

Configuration & validation

  • paths.json — project_root, downloads_root
  • spec_schema.json — JSON Schema v1.0.0
  • manifest_schema.json — manifest schema

3. The 5 AST Transforms

1. ArithmeticTransform

Converts between compound and expanded assignments

x += 1;
x = x + 1;
Bidirectional Low risk
2. SwapCondition

Swaps operands of comparison expressions

a < b
b > a
Semantics-preserving Fix C applied
3. PointerArithmeticToArrayIndex

Converts between pointer arithmetic and array indexing

*(arr + i)
arr[i]
Bidirectional Fix A+B applied
4. TypedefExpansion

Expands typedef aliases to underlying types

typedef float Real;
Real x;
typedef float Real;
float x;
Bug D: struct pointers
5. ChangeNames (vars + functions)

Renames local variables, parameters, and functions consistently

int count = 0;
count++;
int v_0 = 0;
v_0++;
Symbol-aware Token fallback for CUDA

6. Augmentation Levels (L0–L4)

Controlled via LEVEL_FRACTIONS in augment_dataset.py. _greedy_valid_subset() ensures non-overlapping edits at every level. All levels stable after Bug A/B/C fixes merged from erel/aug.

Level Transform Selection Candidate Selection Effective Coverage
L0 None (original source) — Baseline — no augmentation
L1 1 random transform from pool 1 candidate per transform Minimal change — often 1 edit site
L2 33% of transform pool 33% of valid candidates Light coverage — a few edits across ~1 transform
L3 66% of transform pool 66% of valid candidates Moderate coverage — multiple transforms, most sites
L4 All transforms All valid candidates Maximum augmentation — every valid edit applied

Batch Runner Flags

  • --suite — required; avoids cross-suite name collisions
  • --direction — e.g. cuda-to-omp
  • --augment-levels 0 1 2 3 4 — separate tasks per level
  • --models — one or more model identifiers
  • --resume — skip existing result files

Results Aggregation

  • analyze_eval.py aggregates by model / direction / kernel / level
  • --write-dashboard emits eval_results_data.js
  • --show-gaps prints missing (kernel, model, direction, level) combos
  • Machine-readable JSON + publication-ready Markdown tables

Three-Tier Timing Model

  • Tier 1: Kernel time only (profiler / omp_get_wtime())
  • Tier 2: Kernel + data transfer time
  • Tier 3: Wall-clock time (includes I/O, malloc)
  • Paper speedup claims use Tier 1; Tier 3 for total cost
