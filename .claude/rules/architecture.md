# Architecture

> Conditional: loads when working in harness, scripts, or c_augmentation. Kept out of always-loaded CLAUDE.md to save tokens.

## Pattern Overview

- **Spec-centric:** Every operation parameterized by a JSON kernel spec
- **Pipeline-oriented:** Sequential stages (build → run → verify) with clear data contracts
- **Result immutability:** Evaluation and augmentation result JSONs are append-only
- **Manifest-driven:** `manifest.jsonl` is global kernel registry (append-only)
- **Multi-suite:** Rodinia, HeCBench, XSBench, RSBench, mixbench

## Layers

### Spec layer
- Purpose: Define kernels, build/run/verify configs, metadata
- Location: `specs/` (206 JSON files), `manifest.jsonl`, `schema/spec_schema.json`
- Contains: identity, provenance, files, implementation, build, run, verification, performance, hardware, baseline_results, metadata
- Depends on: nothing (pure data)
- Used by: harness, evaluation, augmentation, analysis

### Harness layer
- Purpose: Compile, execute, verify benchmark kernels from specs
- Location: `harness/` (7 Python modules)
- Contents:
  - `spec_loader.py` — load specs, resolve paths, extract prompt payloads
  - `builder.py` — compile via subprocess
  - `runner.py` — execute with optional GNU time
  - `verifier.py` — exit_code + stdout_pattern + stdout_exclude_pattern + numeric_comparison + file_hash strategies
  - `reporter.py` — format results
  - `models.py` — dataclasses (Status, BuildResult, RunResult, VerificationResult, MetricResult, SpecResult)
  - `cli.py` — argparse CLI entry point
- Depends on: spec layer, benchmark source, system compilers (nvcc, gcc, nvc)
- Used by: LLM evaluation, augmentation verification, manual spec testing

### Evaluation layer
- Purpose: Send source to LLMs for translation, grade via harness
- Location: `scripts/evaluation/` (5 Python modules)
- Contents:
  - `llm_evaluate.py` — single-task LLM call + build/run/verify + iterative repair
  - `run_eval_batch.py` — batch orchestrator building task lists from manifest
  - `analyze_eval.py` — result aggregation into summaries
  - `reverify_pass_results.py` — re-check existing PASS results
- Depends on: harness, LLM API clients (anthropic, openai SDKs), specs
- Used by: batch shell scripts in `scripts/batch/`, analysis scripts

### Augmentation layer
- Purpose: Semantics-preserving C/C++ transforms before LLM translation
- Location: `c_augmentation/` (core transforms), `scripts/augmentation/` (pipeline scripts)
- Contents:
  - `augment_dataset.py` — transform classes using libclang AST
  - `test_transforms.py` — unit tests
  - `validate_augmentation.py` — validation
  - `augment_verify.py` — augment → build → verify pipeline
  - `run_augment_batch.py` — batch runner
- Depends on: libclang bindings, harness (for verification)
- Used by: evaluation (via `augment_level`), standalone augmentation verification

### Analysis layer
- Purpose: Aggregate results into tables/figures/stats for paper
- Location: `scripts/analysis/` (20+ modules), `scripts/generate_paper_figures.py`
- Contents: quantitative_findings, statistical_analysis, build_error_taxonomy, benchmark_characterization, cross_consistency_audit, token_analysis, augmentation_analysis, sloc_analysis, generate_paper_data, cross_model_comparison (+ test files)
- Depends on: result JSONs in `results/evaluation/`, specs, manifest
- Used by: paper LaTeX, visualization dashboard

### Visualization layer
- Purpose: Interactive HTML dashboard
- Location: `visualizations/` (HTML pages + JS data files)
- Contents: overview, results, build_results, llm_evaluation, pipeline, architecture, augmentation_deep_dive, benchmark_landscape, sprint_dashboard; data files (`results_data.js`, `eval_results_data.js`, `build_results_data.js`)
- Depends on: data from `scripts/generate_viz_data.py` and `analyze_eval.py --write-dashboard`
- Used by: GitHub Pages deployment

## Data Flow

- No runtime state database; all state lives in JSON files on disk
- Evaluation state via file existence: `--resume` skips tasks with existing result JSONs
- Manifest is append-only: deleted specs leave orphan manifest entries (expected)
- Baseline results stored directly in spec JSONs (`baseline_results` field)

## Key Abstractions

### Kernel Spec
- Complete definition of one benchmark kernel variant
- Examples: `specs/rodinia-bfs-cuda.json`, `specs/hecbench-nn-omp.json`
- JSON conforming to `schema/spec_schema.json` (version `1.0.0`)

### Translation Pair
- `(source_spec, target_spec)` sharing kernel_name, differing `parallel_api` (e.g., rodinia-bfs-cuda → rodinia-bfs-omp)
- Generated by `find_translation_pairs()` in `harness/spec_loader.py`

### Translation Targets
- Subset of `files.prompt_payload` that the LLM must produce (kernel-centric translation)
- Field: `files.translation_targets` in every spec
- Families: OpenCL targets = `.cl` files only (kernel-only); OMP/CUDA = curated kernel files

### Evaluation Result
- Per-task JSON: overall_status, source_spec, target_spec, model, augment_level, attempts[], prompt_tokens, completion_tokens, timing, translated code snippets
- Statuses: PASS, BUILD_FAIL, RUN_FAIL, VERIFY_FAIL, EXTRACTION_FAIL, ERROR, SKIP, TIMEOUT

### Status (enum)
- PASS, FAIL, ERROR, TIMEOUT, SKIP
- Used by BuildResult, RunResult, VerificationResult

### Verification Strategy
- Implemented: `exit_code`, `stdout_pattern`, `stdout_exclude_pattern`, `numeric_comparison`, `file_hash`
- Stub: `file_diff`, `custom_script`
- Semantics: conjunction — ALL non-SKIP strategies must PASS for overall PASS

### AstTransform
- ABC subclass with `discover()` + `apply()` using libclang cursors
- Examples: `ArithmeticTransform`, `SwapCondition`, `PointerArithmeticToArrayIndex`, `TypedefExpansion`, `ChangeNames`, `ChangeFunctionNames`

## Entry Points

### Harness CLI
- `harness/__main__.py` → `harness/cli.py:main()`
- Commands: `build`, `run`, `verify`, `prompt`, `info`, `pairs`
- **Critical:** `-v` flag must come BEFORE subcommand

### Batch Evaluation
- `scripts/evaluation/run_eval_batch.py`
- Invoke: `python3 scripts/evaluation/run_eval_batch.py --suite rodinia --direction cuda-to-omp --models <model> --project-root <path> --resume -v`

### Single-Task Evaluation
- `scripts/evaluation/llm_evaluate.py` — direct CLI or imported by batch runner

### Result Aggregation
- `scripts/evaluation/analyze_eval.py --project-root <path>` → `eval_summary.{json,md}`

### Schema Validation
- `scripts/validate_schema.py` — pre-commit and CI

### Augmentation
- `scripts/augmentation/augment_verify.py` — apply transforms then build/run/verify

### Paper Figures
- `scripts/generate_paper_figures.py` — matplotlib figures for LaTeX

### Batch Shell Scripts
- `scripts/batch/` — `run_eval_campaign.sh`, `run_rodinia_batch.sh`, `run_cuda_batch.sh`, `run_omp_batch.sh`; archived: `_archive/pre-phase3-2026-03-16/run_xsbench_eval.sh`, `run_rodinia_augmented_eval.sh`

## Error Handling

- Harness ops return typed result objects with Status enum — never raise for expected failures
- `evaluate_translation()` uses try/finally for file restoration (backup/restore pattern)
- Build errors captured as head+tail snippets (`_head_tail()`) for retry prompts and result JSONs
- LLM API failures caught → stored in result JSON `error_message`
- Iterative repair: on BUILD_FAIL, `analyze_build_failure()` parses linker errors, finds missing symbols, generates targeted repair hints for retry
- Subprocess timeouts configurable (default 300s run, 600s build)
