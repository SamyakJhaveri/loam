## Common Pitfalls

### Pitfall 1: Suite Filter Removal Without Direction Check
**What goes wrong:** Removing `suite="rodinia"` from F3/F4 without also confirming `direction` filter excludes omp_target results in ALL cells.
**Why it happens:** HeCBench has omp_target directions. Without direction filtering, omp_target results contaminate standard-direction figures.
**How to avoid:** Always filter by `direction in DIRECTIONS` (the canonical 6-direction list) whenever removing suite filters. The `DIRECTIONS` constant already excludes omp_target.
**Warning signs:** Figure shows more tasks/kernels than expected; HeCBench omp_target-only kernels appear.

### Pitfall 2: Heatmap Aspect Ratio for 29 Kernels
**What goes wrong:** 29-row heatmap with default `figsize=(7.16, 3.5)` produces unreadable tiny cells.
**Why it happens:** Original heatmap was 18 kernels x few models. Scaling to 29 kernels needs taller figure.
**How to avoid:** Compute figure height dynamically: `height = max(4, n_kernels * 0.3 + 2)`. Use `fontsize=7` for kernel labels with 29+ rows.
**Warning signs:** Kernel labels overlap; cell text unreadable; matplotlib tight_layout warnings.

### Pitfall 3: GPT-4.1 Mini Column With No Data
**What goes wrong:** GPT-4.1 mini model directory doesn't exist yet. Code crashes trying to load data.
**Why it happens:** D-01 requires GPT-4.1 mini columns but no data files exist on disk.
**How to avoid:** GPT-4.1 mini must be rendered as static N/A cells, not loaded from data. Add the model to display constants but handle missing data gracefully in each figure function (render gray cells, "pending" labels, etc.).
**Warning signs:** `FileNotFoundError` or empty results for GPT-4.1 mini.

### Pitfall 4: build_kernel_model_matrix Hardcoded suite="rodinia"
**What goes wrong:** Default `suite` parameter in `build_kernel_model_matrix()` is `"rodinia"` (line 428). Callers that don't override this will still get Rodinia-only data.
**Why it happens:** Function signature was designed for single-suite usage.
**How to avoid:** Change default to `None` (all suites) or pass explicit `suite=None` at all call sites. Verify every call site.
**Warning signs:** Figures show only 18-21 kernels despite code claiming "all suites".

### Pitfall 5: Stale Fallback Data
**What goes wrong:** Script falls back to `AUG_ROBUSTNESS` or `XSBENCH_L0` dictionaries containing old 3-model data.
**Why it happens:** Fallback branches triggered when result files aren't found or data is empty.
**How to avoid:** Per D-02, remove all old fallback data dictionaries entirely. The 1,248 Qwen result files exist and load_eval_results() works. No fallback needed.
**Warning signs:** Figure shows old model names (Claude, Gemini, Groq).

### Pitfall 6: HeCBench Kernel Names in Heatmap
**What goes wrong:** HeCBench kernel names like "floydwarshall" collide with hypothetical future Rodinia additions, or are ambiguous without suite context.
**Why it happens:** Kernel names are stored without suite prefix in result JSONs.
**How to avoid:** In the heatmap, group by suite with visual separators. The kernel name within each suite group is unambiguous. No kernel name collision exists in current data (verified: 34 unique names across all suites).
**Warning signs:** Duplicate kernel rows in heatmap.
