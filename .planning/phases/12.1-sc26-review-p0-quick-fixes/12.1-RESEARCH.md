### Fix 3: SIMT/SPMD Correction (line 509)

**Current (WRONG):**
```latex
Execution model & SPMD (warps of 32 threads) & ...
```

**Should be:**
```latex
Execution model & SIMT (warps of 32 threads) & ...
```

CUDA's execution model is SIMT (Single Instruction, Multiple Thread), not SPMD (Single Program, Multiple Data). SPMD describes the programming model at a higher level (all threads run the same kernel code), but the hardware execution model is SIMT -- threads within a warp execute in lockstep, with divergent branches serialized. The NVIDIA CUDA Programming Guide explicitly uses "SIMT" to describe the warp-based execution model. [VERIFIED: NVIDIA CUDA Programming Guide Section on SIMT Architecture; also consistent with CLAUDE.md mentoring rule on warp divergence]

This is a single-cell replacement in Table 3. No other references to "SPMD" exist in the paper context that need changing.

## Common Pitfalls

### Pitfall 1: Changing Other Benchmarks' pass@1 References
**What goes wrong:** Blindly replacing all "pass@1" with "greedy pass rate" changes citations to other benchmarks (ParEval-Repo, HPC-Coder-V2) which legitimately use "pass@1".
**Why it happens:** "pass@1" appears 15+ times in the paper; only ~5 refer to ParBench's metric.
**How to avoid:** Use the Fix 4 table above. Only change lines where the subject is ParBench's primary campaign metric. Lines referencing other benchmarks or the T=0.7 pass@k experiment stay as-is.
**Warning signs:** If a "pass@1" reference is preceded by a citation like `\cite{ParEvalRepo2025}`, it should NOT be changed.

### Pitfall 2: Inconsistent Provenance Comments
**What goes wrong:** Updating the prose number but leaving the provenance comment pointing to `quantitative_findings.json`.
**Why it happens:** Provenance comments are easy to miss when making targeted edits.
**How to avoid:** For Fix 2 (lines 333, 388), update BOTH the comment AND the prose in the same edit.
**Warning signs:** `grep "quantitative_findings" paper.tex` should return 0 matches after the fix (Phase 12 already updated most; these are the last 2).

### Pitfall 3: Table Cell Width Overflow in Table 3
**What goes wrong:** The new CUDA source structure text is too long for the `p{4.0cm}` column, causing awkward line breaks or overflow.
**Why it happens:** The original text "Single file (host + device code interleaved)" was already near the column width limit.
**How to avoid:** Test the LaTeX output (or estimate character width) before committing. Keep the replacement text concise.
**Warning signs:** If the replacement text is significantly longer than the original, consider abbreviating.

### Pitfall 4: Forgetting the Table 1 Footnote LaTeX Syntax
**What goes wrong:** Using `\footnote{}` inside `tabular*` which may place the note at the page bottom rather than under the table.
**Why it happens:** IEEEtran has specific footnote behavior inside floats.
**How to avoid:** Use a `\par\vspace{2pt}\footnotesize` note between `\end{tabular}` and `\end{table*}`, or add a `\multicolumn` row. Both are standard IEEE patterns.
**Warning signs:** The footnote appearing at the bottom of the page column instead of directly under the table.

### Pitfall 5: Breaking the pass@k Table Alignment
**What goes wrong:** Renaming the "pass@1 (greedy, T=0)" column header in tab:pass-at-k changes the column width and misaligns the table.
**Why it happens:** LaTeX table column widths are sensitive to content length.
**How to avoid:** "Greedy pass rate (T=0)" is similar in length to "pass@1 (greedy, T=0)" so alignment should be stable. Verify after editing.
