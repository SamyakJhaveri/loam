## Implementation Decisions

### Table 1 Presentation (Fix 1)
- **D-01:** Remove "~2,500 eval tasks" and "2 models" from Table 1's Scale cell. Replace with benchmark-intrinsic metrics: "96 specs, 35 kernels, 5 suites, 8 directions".
- **D-02:** Add a table footnote for campaign details: "Initial campaign: 1,136 tasks with Qwen 3.5 397B-A17B; GPT-4.1 mini campaign in progress." Task count (1,136) is campaign-specific, not benchmark-intrinsic, so it belongs in the footnote rather than the cell.

### pass@1 Terminology (Fix 4)
- **D-03:** Rename "greedy-decode pass@1" to "greedy pass rate" everywhere it refers to ParBench's primary metric (T=0, up to 3 self-repair retries). The term "pass@1" is misleading for a metric with retries since it conventionally means single-shot (Chen et al., HumanEval).
- **D-04:** Reserve "pass@k" exclusively for the T=0.7 sampling experiment (no self-repair, k=1,3). This creates a clean separation: "greedy pass rate" = deterministic with retries; "pass@k" = stochastic without retries.
- **D-05:** Leave other benchmarks' pass@1 references untouched (e.g., HPC-Coder-V2's pass@1 on ParEval at line ~221). Only ParBench-specific terminology changes.
- **D-06:** Update the definition paragraph at line ~700 to define "greedy pass rate" instead of "greedy-decode pass@1". Preserve the full explanation of what the metric includes (T=0, 3 retries, build+run+verify).

### Table 3 Source Structure (Fix 5)
- **D-07:** Update CUDA "Source structure" cell from "Single file (host + device code interleaved)" to "Single or multi-file (host + device code interleaved or split across files; 51% multi-file in corpus)". This acknowledges both the general API property and ParBench's specific corpus reality.

### 700→710 Inconsistency (Fix 2)
- **D-08:** Update lines ~333 and ~388 from 700 to 710 denominators. Change provenance comments to reference `paper_data.json` instead of `quantitative_findings.json`. Line 333 (S3.2): VERIFY_FAIL = 51/710; Line 388 (S3.4): BUILD_FAIL = 241/710. These are the only two remaining stale references.

### SIMT/SPMD (Fix 3)
- **D-09:** Change line ~509 Table 3 CUDA execution model from "SPMD (warps of 32 threads)" to "SIMT (warps of 32 threads)". CUDA's execution model is Single Instruction Multiple Thread, not Single Program Multiple Data.

### Claude's Discretion
- Exact LaTeX formatting of the Table 1 footnote (placement, symbol choice)
- Wording adjustments to surrounding prose when "greedy pass rate" replaces "greedy-decode pass@1" in narrative sentences
- Whether to update the table header in the pass@k table (line ~1077) that currently says "pass@1 (greedy, T=0)"
