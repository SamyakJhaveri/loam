```latex
\paragraph{Case Study 1: CFD Euler Flux (CUDA-to-OpenCL).}
% src: error_taxonomy.json > verify_fail > cfd; 16 VERIFY_FAIL instances
The \texttt{cfd} kernel computes Euler flux across an unstructured mesh.
In 16 of 18 CUDA-to-OpenCL translations, the LLM produced code that
compiled and ran but produced incorrect numerical output. Manual
inspection reveals incorrect reduction scope: the flux accumulation
across neighboring cells uses a local reduction variable that is not
properly scoped to the work-item, causing partial sums to overwrite
each other. This represents a \emph{reduction scope error}---the
LLM preserves the loop structure but fails to translate CUDA's
implicit per-thread accumulation to OpenCL's explicit work-item model.

\paragraph{Case Study 2: Hotspot 2D Stencil (OpenCL-to-OMP).}
% src: error_taxonomy.json > verify_fail > hotspot; 7 VERIFY_FAIL instances
The \texttt{hotspot} kernel performs a 2D iterative stencil computation
on a temperature grid. In 7 OpenCL-to-OMP translations, the code
compiles and executes but produces numerically divergent output.
The root cause is incorrect \emph{thread mapping}: the LLM collapses
OpenCL's 2D work-group indexing (\texttt{get\_global\_id(0/1)}) into
a single OpenMP parallel-for loop without preserving the row-major
iteration order, introducing data races on boundary elements.

\paragraph{Case Study 3: Gaussian Elimination (OpenCL-to-CUDA).}
% src: error_taxonomy.json > verify_fail > gaussian; 6 VERIFY_FAIL instances
The \texttt{gaussian} kernel performs row elimination with explicit
synchronization. In 6 OpenCL-to-CUDA translations, the LLM fails
to translate the \emph{memory model} correctly: OpenCL's
\texttt{barrier(CLK\_LOCAL\_MEM\_FENCE)} maps to CUDA's
\texttt{\_\_syncthreads()}, but the LLM places the synchronization
outside the reduction loop, allowing threads to read partially
updated shared memory. This is a memory model reasoning failure,
not a syntax error.
```
