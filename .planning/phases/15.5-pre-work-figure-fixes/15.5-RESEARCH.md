### Pattern: Dual-Model Line Chart (F7 Refactoring)

**What:** Replace the single-model aggregate loop with a per-model iteration over `MODEL_COLORS.keys()`. [VERIFIED: codebase analysis]

**Current code (lines 1140-1150):**
```python
# Aggregate pass counts and totals per level (single model: Qwen)
level_pass: dict[int, int] = defaultdict(int)
level_total: dict[int, int] = defaultdict(int)
for r in c2o_all:
    lvl = r.get("augment_level", 0)
    level_total[lvl] += 1
    if r.get("overall_status") == "PASS":
        level_pass[lvl] += 1
```

**Refactored pattern:**
```python
# Split by model, compute per-model pass rates and CIs
for model_id in MODEL_COLORS:
    model_recs = [r for r in c2o_all if r["model"] == model_id]
    if not model_recs:
        continue
    
    level_pass = defaultdict(int)
    level_total = defaultdict(int)
    for r in model_recs:
        lvl = r.get("augment_level", 0)
        level_total[lvl] += 1
        if r.get("overall_status") == "PASS":
            level_pass[lvl] += 1
    
    pass_counts = [level_pass.get(lvl, 0) for lvl in levels]
    rates = [c / max(level_total.get(lvl, 1), 1) * 100 for lvl, c in zip(levels, pass_counts)]
    
    # Wilson CIs for error bars (D-07)
    ci_low = [_wilson_ci(level_pass.get(lvl, 0), level_total.get(lvl, 1))[0] * 100 for lvl in levels]
    ci_high = [_wilson_ci(level_pass.get(lvl, 0), level_total.get(lvl, 1))[1] * 100 for lvl in levels]
    yerr_low = [r - lo for r, lo in zip(rates, ci_low)]
    yerr_high = [hi - r for r, hi in zip(rates, ci_high)]
    
    color = MODEL_COLORS[model_id]
    label = MODEL_DISPLAY_SHORT[model_id]
    marker_fmt, linestyle = MODEL_LINESTYLE[model_id]
    
    ax.errorbar(
        levels, rates,
        yerr=[yerr_low, yerr_high],
        fmt=marker_fmt,
        color=color,
        linewidth=1.8,
        markersize=7,
        label=label,
        capsize=3,
        capthick=1.0,
    )
```

### Pattern: Error Bars with `ax.errorbar()` Instead of `ax.plot()`

**What:** Replace `ax.plot()` with `ax.errorbar()` to support CI bars in the same call. [CITED: matplotlib 3.10 docs -- `Axes.errorbar()`]

**Key difference:** `ax.errorbar()` takes a `fmt` parameter (e.g., `"D-."`) instead of separate format string in positional args. The `yerr` parameter accepts `[lower_errors, upper_errors]` for asymmetric error bars. This matches the Wilson CI (asymmetric by nature).

**When to use:** Always when both data points and error bars are needed on the same line.

### Pattern: Per-Point Annotation Adjustment

**What:** The current per-point annotation (lines 1169-1177) needs to account for two lines at different y-positions. Annotate below each line with the model's color for visual grouping. [VERIFIED: codebase]

**Consideration:** With two lines, annotations may overlap. The Qwen line averages ~59% and GPT ~37% -- there is a ~22pp gap, which provides sufficient vertical separation for annotations at most levels. However, the planner should consider making annotation optional or reducing font size if overlap occurs at any level.

## Common Pitfalls

### Pitfall 1: GPT Result JSONs Have `direction: None`
**What goes wrong:** If code reads `direction` from the raw JSON instead of the computed record, GPT records will be filtered out.
**Why it happens:** GPT eval results were generated by a collaborator; the `direction` field was not populated in the JSON.
**How to avoid:** The `load_eval_results()` function computes `direction` from `source_spec` and `target_spec` at line 346. Always use the record dict, never the raw JSON. [VERIFIED: runtime test confirmed GPT records have computed `direction = "cuda-to-omp"` in records]
**Warning signs:** F7 shows only one line (Qwen), GPT line missing.

### Pitfall 2: `MODEL_LINESTYLE` Format Mismatch with `errorbar()`
**What goes wrong:** `MODEL_LINESTYLE` stores `("D-.", "dashdot")` for Qwen and `("v-", "solid")` for GPT. The first element is a combined marker+line format string. `ax.errorbar()` uses the `fmt` parameter for this.
**Why it happens:** `ax.plot()` accepts the format string as the third positional argument; `ax.errorbar()` uses `fmt=` keyword.
**How to avoid:** Pass `fmt=MODEL_LINESTYLE[model_id][0]` explicitly. [VERIFIED: codebase line 99-101]
**Warning signs:** Compilation error or missing markers/line styles.

### Pitfall 3: Annotation Overlap with Two Lines
**What goes wrong:** Per-point rate annotations (e.g., "61.5%") from Qwen and GPT may overlap, especially if rates converge at certain levels.
**Why it happens:** Both annotation sets use `xytext = (lvl, rate - 5)` offset.
**How to avoid:** Annotate Qwen below (`rate - 5`) and GPT above (`rate + 3`), or skip annotations entirely and rely on the legend + error bars for readability. Alternatively, only annotate L0 and L4 endpoints.
**Warning signs:** Visually cluttered figure with overlapping text.

### Pitfall 4: Y-Axis Limits May Need Adjustment
**What goes wrong:** Current `ax.set_ylim(-2, 85)` was tuned for Qwen ~60% data only. GPT data at ~37% fits within this range, but with two lines, the visual may benefit from adjustment.
**Why it happens:** Original ylim was set for a single model's range.
**How to avoid:** Keep `(-2, 85)` -- both models fit. The extra vertical space helps separate the annotation text. [VERIFIED: GPT max is 42.3% at L4, Qwen max is 65.4% at L2 -- both within 85%]
**Warning signs:** Lines appear cramped at the bottom of the plot.

### Pitfall 5: `aug_total` Variable Used in Title
**What goes wrong:** Line 1148 sets `aug_total = level_total.get(0, 1)` which in the current single-model code gives the count for a single model. After refactoring to dual-model, this needs to be set to N=26 (per model) not N=52 (total).
**Why it happens:** The title should say "N=26 kernels" (per model, symmetric), not 52.
**How to avoid:** Compute `n_kernels` from the first model's L0 count, or hard-derive from the data. The title per D-08 should state "N=26 cuda-to-omp kernels per level, both models". [VERIFIED: CONTEXT.md D-08]
**Warning signs:** Title says "52 kernels" which misrepresents the sample size.
